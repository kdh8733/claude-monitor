## 배경·전제 (뒤집지 않는다)

- 언어/런타임: Node 24 / TypeScript 단일. 저장: `node:sqlite`(WAL). 웹: Next.js App Router. 차트: visx. UI: shadcn/ui + Tailwind v4. (전부 001에서 승인.)
- 실시간 스트리밍은 Non-goal. 배치다.
- 한도의 권위 있는 출처는 `/api/oauth/usage` 스냅샷뿐. 트랜스크립트는 "어디에 썼나"만 답한다.
- **계약 경계는 SQLite 파일 하나다.** 쓰기 측(수집 데몬)이 스키마 DDL과 불변식을 확정한다. 이 문서는 **그 파일을 읽기 전용으로 여는 쪽**을 설계하고, 쓰기 측에 요구할 사항을 마지막 절에 목록화한다. 스키마를 여기서 발명하지 않는다.
- 검증된 사실(이 문서에서 활용):
  - **`node:sqlite`는 `readOnly` 옵션을 지원**하며, **파일이 없으면 열기가 실패한다.** 공식 문서 인용: *"If `true`, the database is opened in read-only mode. If the database does not exist, opening it will fail. Default: false."* (Node v24 `node:sqlite` docs). 이 "없으면 실패"가 배포 안전성의 방벽으로 쓰인다(축 3).
  - **Node 24는 타입 스트리핑을 기본 지원**한다: *"Node 24 ships with native type stripping enabled out of the box, so `node app.ts` Just Works. No tsconfig.json required."* 단, **tsconfig를 읽지 않으며**(paths 등 미지원), **enum/namespace는 `--experimental-transform-types` opt-in**이 필요하고, **타입 검사는 하지 않는다**(amaro가 스트립만). (Node.js Modules: TypeScript docs.)
  - **Next.js `output: export`는 요청 시점 동적 로직을 지원하지 않는다**: *"Features that require a Node.js server, or dynamic logic that cannot be computed during the build process, are not supported."* (Next.js Static Exports guide).

## 축 1. 집계 계층의 위치

판정 기준: 완료 기준 3(귀속 정확도 - 독립 재계산 스크립트와 오차 0)을 테스트하기 쉬운 구조인가. 단가 개정(시점 버전) 시 과거 집계가 어떻게 되는가.

| 항목 | (a) SQL view/쿼리 | (b) 적재 시 materialized | (c) 요청 시 TS가 raw행 계산 | (d) SQL+TS 혼합 (권고) |
|---|---|---|---|---|
| 토큰 집계(project/model/time GROUP BY) | SQL이 최적 | 미리 저장 | JS로 전량 로드 | **SQL** |
| 시점 버전 단가 적용 | 범위 조인(까다로움) | 저장 시 못 다룸 | TS map | **TS(읽기전용 제약상 불가피)** |
| 단가 개정 시 과거 집계 | 재계산이 곧 재조회(무비용) | **전 구간 backfill 필요** | 재조회 | 재조회(무비용) |
| 완료기준3 오라클 독립성 | 원본 JSONL 재계산과 데이터경로 분리 | 저장값과 비교 → 약함 | 같은 언어라 코드 공유 유혹 | 원본 JSONL 재계산과 분리 |
| 쓰기/읽기 경계 오염 | 없음 | **읽기 로직이 write측으로 샘** | 없음 | 없음 |

**왜 가장 단순한 안(a 순수 SQL)으로 충분하지 않은가:** 순수 SQL이 이 규모(연 ~20만 행)에서 성능·단순성 모두 이상적이지만, **읽기 측이 DB를 읽기 전용으로 연다**(검증됨). 그래서 시점 버전 단가표(축 2)를 읽기 측이 메인 DB에 조인 테이블로 심을 수 없다. 단가 적용 한 단계만은 TS로 나가야 한다 → 결과적으로 (d) 혼합이 된다. 순수 SQL(a)은 "쓰기 측이 단가 테이블을 메인 DB에 시딩"을 전제해야 성립하는데, 이는 계약 경계를 넓히고 단가 개정마다 재시딩을 강제한다.

- **권고: (d) SQL+TS 혼합, 요청 시 계산.** 분할선은 원칙적이다 - **SQL = 집합 집계**(토큰 합, `weekly_all` 구간 평균, `json_extract`로 스코프 순위), **TS = 시점 버전 단가 적용 + 헤드룸 성형 + 응답 조립.** materialized 없음 → 단가 개정은 재조회일 뿐 backfill이 없다(CLAUDE.md 5 자동 충족).
- **차선: (a) 순수 SQL.** 쓰기 측이 단가 테이블을 메인 DB에 시딩해 준다면 채택 가능. 단가 개정마다 재시딩이 필요.
- **기각: (b) materialized** - 단가 개정 시 전 구간 backfill, 읽기 로직이 쓰기 측으로 새어 계약 오염, 현 규모에 과설계. **(c) 순수 TS-raw** - GROUP BY를 JS로 대체해 대량 행을 프로세스에 끌어올림, 집합 집계는 SQL의 일이다.

**완료기준3 테스트 구조:** 프로덕션 경로 = SQLite 이벤트 테이블 → SQL 집계 → TS 단가. 오라클 = **원본 JSONL을 직접 재계산하는 독립 스크립트**. 독립성은 데이터 경로 분리(DB vs 원본 파일)에서 나온다. 오차 0 단언을 TDD로 먼저 쓴다(ROADMAP M5).

## 축 2. 단가표 관리

요건: `effective_from` 보유, 과거 이벤트는 당시 단가로 계산, 변경 이력 감사 가능. **단가는 공개 정보라 커밋 가능**(실데이터가 아님).

| 항목 | (a) TS 상수 배열 (권고) | (b) DB 테이블 시딩 | (c) 커밋 JSON/YAML 런타임 로드 |
|---|---|---|---|
| `effective_from` | 타입 있는 필드 | 컬럼 | 필드 |
| 감사(변경 이력) | **git diff = 감사 로그** | 행 버전 없으면 이력 없음 | git diff = 감사 로그 |
| 타입 안전 | **컴파일 타임** | 없음 | 로드 후 검증 필요 |
| 읽기전용 DB 제약 | 무관(TS가 소유) | **읽기 측이 못 씀** | 무관 |
| 실패 표면 | 없음(IO 없음) | - | 파싱/검증 실패 경로 추가 |

**왜 가장 단순한 안(a)으로 충분한가:** 세 요건(effective_from·소급 정확·감사)을 모두 충족한다. (c) 대비 유일한 약점은 "재빌드 없이 편집"인데, 1인·TS-전면 프로젝트에서 단가 수정은 어차피 TS를 만지는 작업이라 실질 제약이 아니다. 단가 적용은 TS에서 일어나므로(축 1의 d) TS 상수가 소비처와 같은 언어다. 그래서 (a)가 가장 단순하면서 충분하다.

- **권고: (a) TS 상수 배열.** `(model, category, effective_from, price)` 형태. git 이력이 감사 로그. 컴파일 타임 타입 검사. 런타임 IO/파싱 실패 없음.
- **차선: (c) 커밋 JSON.** 단가가 커지거나(다수 tier), 비개발자가 편집해야 하면 승격. 로드 시 스키마 검증 추가.
- **기각: (b) DB 테이블 단독** - 행 버전 없이는 감사 이력이 없고, 읽기 측이 읽기전용 DB에 쓸 수 없다.

**단가표 초안 (출처: <https://platform.claude.com/docs/en/about-claude/pricing.md>, 2026-07-09 조회. USD/MTok).** ROADMAP 모델 분포(opus-4-8, fable-5, haiku-4-5, sonnet-5, sonnet-4-6)를 포함. 트랜스크립트에 캐시 토큰(`cache_read_input_tokens`, `cache_creation.ephemeral_{5m,1h}`)이 있으므로 캐시 카테고리도 싣는다.

| model | effective_from | input | output | cache_write_5m | cache_write_1h | cache_read |
|---|---|---|---|---|---|---|
| claude-opus-4-8 | (최초 관측가) | 5 | 25 | 6.25 | 10 | 0.50 |
| claude-fable-5 | (최초 관측가) | 10 | 50 | 12.50 | 20 | 1.00 |
| claude-haiku-4-5 | (최초 관측가) | 1 | 5 | 1.25 | 2 | 0.10 |

- **미확인:** 프로젝트 데이터 구간은 2026-05-08부터인데, **2026-07-09 이전 시점의 소급 단가는 조사로 확정하지 못했다.** 위 값은 조회 시점의 현재가다. `effective_from`을 현재가에 붙일 때 "최초 관측가"의 실제 발효일을 확정하려면 모델별 가격 개정 이력을 추가 조사해야 한다. 이 미확인은 메커니즘(축 2 채택)과 무관하며, 표 값의 발효일만 후속 보정 대상이다.

## 축 3. 로컬(실데이터) / Vercel(데모) 단일 코드베이스 스위칭

판정 기준: 완료 기준 6(공개 레포 실데이터 0바이트, 배포 네트워크 응답 실데이터 0). **실수로 실데이터가 빌드에 섞이는 것을 구조적으로 불가능하게** 만드는 안 최우선.

| 항목 | (a) env 어댑터 교체 | (b) 빌드타임 시드 bake + `output: export` (권고) | (c) 두 개 앱 | (d) 파일 경로만 교체 |
|---|---|---|---|---|
| 배포 런타임 | 서버 존재 | **정적, 서버 없음** | 서버/정적 | 서버 |
| 데모 데이터 입력 | 어댑터가 실 or 데모 | **커밋된 시드 JSON만** | 앱별 | 경로 지정 |
| 실데이터 빌드 혼입 | 설정 오류로 가능성 | **구조적 차단** | 분리되나 중복 | gitignore가 데모 sqlite 금지→성립불가 |
| 1인 운영 부담 | 낮음 | 낮음 | **높음(중복)** | 낮음 |

**구조적 안전성의 층 (b 채택 근거):**
1. **1차 방벽 - gitignore:** 실데이터는 `data/`, `*.sqlite`(+wal/shm)에만 존재하고 gitignore로 차단(CLAUDE.md 1, 완화 금지). Vercel은 git 체크아웃에서 빌드 → **실데이터는 빌드 환경에 물리적으로 부재.**
2. **2차 방벽 - `output: export`:** 배포물은 커밋 시드에서 빌드타임에 구운 정적 파일. **런타임 데이터 접근 자체가 없다**(검증: export는 요청 시점 동적 로직 미지원). 경로가 틀려도 요청 시 실데이터를 읽을 서버가 없다.
3. **3차 방벽 - fail-loud 읽기전용 열기:** 만약 데모 빌드가 실수로 실 DB 경로를 참조해도, Vercel엔 그 파일이 없고 **`readOnly`는 없는 파일을 열면 실패**(검증됨) → 조용한 유출이 아니라 **빌드 에러**로 터진다.
4. **4차 방벽 - 카나리 grep:** 빌드 산출물 `out/`을 알려진 실 프로젝트명·프롬프트 지문 문자열로 grep해 CI에서 실패시킨다(완료기준6 검증 절차와 동일).

**왜 (d)가 성립 불가한가:** "데모 sqlite 경로로 교체"는 데모 sqlite를 커밋해야 하는데, `*.sqlite`는 gitignore로 차단되고 **CLAUDE.md 1이 이 완화를 금지**한다. 데모 시드는 sqlite가 아니라 커밋 가능한 JSON이어야 한다(축 5). 그래서 (d)는 규칙에 막힌다.

- **권고: (b) 커밋 시드에서 빌드타임 bake + `output: export`.** 로컬은 동적 렌더(RSC가 실 DB를 읽기전용으로 열어 집계만 렌더), 데모/Vercel은 정적 export(시드만). 데이터 접근은 프로바이더 심(축 6/1)으로 추상화하되, **안전성은 위 4층에서 나온다.**
- **차선: (a) env 어댑터 + 동적 Vercel 배포.** 나중에 라이브성 데모/ISR이 필요하면. 서버가 남아 "0 실데이터" 보장이 약해지므로 요건이 바뀔 때만.
- **기각: (c) 두 앱**(중복, 1인 부담), **(d) 경로 교체**(gitignore/CLAUDE.md 1에 막힘, 구조적 안전성 없음).

**데모 데이터 경로 (축 1/5와 통합):** 데모 빌드는 커밋 시드 JSON을 **빌드타임에 임시 합성 sqlite로 적재 → 로컬과 동일한 SQL 집계 재사용**을 권한다. 단일 집계 코드 경로라 완료기준5("시드로 동일 화면 재현")가 거의 공짜로 성립한다. 임시 DB는 합성 데이터로만 만들어지고(빌드 환경 임시 파일, `.sqlite`로 커밋 안 됨) 실 DB는 부재하므로 안전. 이 빌드타임 sqlite 생성이 부담스러우면 **대안: 데모 프로바이더가 JSON을 TS로 직접 집계**(두 집계 구현 드리프트 위험) - 구현 시 택일.

## 축 4. 코드 배치 / 빌드

판정 기준: 타입 공유, collector가 Next 의존성을 안 끌고 오는가, cron 엔트리포인트 단순함. **모노레포는 정당화 필요.**

| 항목 | (a) 단일 패키지 | (b) npm workspaces 모노레포 (권고) | (c) 별도 레포 |
|---|---|---|---|
| 타입 공유(계약) | import로 자명 | **`shared` 패키지가 계약면** | 레포 간 동기화 필요 |
| collector ⊥ Next | **불가**(한 package.json) | collector→shared만 | 분리되나 계약 분산 |
| cron 엔트리포인트 | Next 트리 포함 install | **최소 트리, 한 파일** | 최소 |
| 배포(Vercel) | collector까지 install | web 패키지만 | 별도 |
| 1인 도구 부담 | 최저 | 낮음(workspace 설정) | **높음(2 레포)** |

**모노레포 정당화 (요구된 대로):** 계약 경계는 **타입**이다 - SQLite 행 형태(스냅샷/이벤트), `/api/oauth/usage` raw JSON 형태, 단가 형태. 쓰기 측(별도 에이전트)과 읽기 측이 이 타입에 합의해야 한다. `shared` 패키지가 이 계약의 자연스러운 집이고, collector와 web 어느 쪽도 상대에 의존하지 않고 import한다. 동시에 **collector는 Next.js를 절대 안 끌고 와야 한다**(판정 기준). 단일 패키지(a)는 collector 의존성을 web에서 격리할 수 없다. 두 요구(의존성 격리 collector⊥Next + 공유 타입 계약면)가 **지금 실재**하므로(collector·web 둘 다 존재) 모노레포는 추측이 아니라 정당하다.

- **권고: (b) 모노레포 3패키지** - `shared`(타입=계약면), `collector`(→shared, Next 없음), `web`(→shared+Next). cron은 `collector`의 엔트리포인트 한 파일을 실행.
- **차선: (a) 단일 패키지** - collector 의존성 발자국이 작게 유지되고 collector install 트리에 Next가 껴도 감수한다면. workspace 설정이 없어 도구가 더 단순.
- **기각: (c) 별도 레포** - 계약이 레포를 가로질러 분산, 공유 타입 동기화 복잡, 단일 공개 레포 모델(하나의 gitignore·배포)이 이미 확립됨. 1인 운영에 과분리.

**TS 실행 방식:**
- **collector 런타임: Node 24 네이티브 타입 스트리핑.** `node collector/src/main.ts`(Node 24 기본, 빌드 산출물 없음) → "cron 엔트리포인트 단순함" 판정 기준 충족. **tsx 불필요**(Node 24가 대체).
- **제약(검증됨) 준수:** ① Node는 tsconfig를 안 읽으므로 paths 별칭 금지 - 상대경로/패키지명으로 해결. ② enum/namespace는 transform-types opt-in이 필요하니 **enum 대신 union type / const object 사용**(strip-only 유지). ③ 타입 스트리핑은 타입 검사를 안 하므로 **`tsc --noEmit`를 CI 게이트로 별도 실행.**
- web: Next.js 자체 TS 파이프라인(SWC/Turbopack) 사용 - node 스트리핑과 무관.
- **검증 못 함(구현 시 확인):** 워크스페이스 `shared` 패키지를 raw `.ts`로 cross-package import할 때 스트립 모드가 그대로 먹는지. Node의 타입 스트리핑이 node_modules(심링크된 워크스페이스 포함) 파일에 항상 적용되는지는 버전별 뉘앙스가 있어 실측 필요. **폴백:** `shared`만 `tsc`로 `.js`+`.d.ts` 빌드해 소비(추측성 단정 회피).

## 쓰기 측에 대한 요구사항 목록 (접합면)

읽기 측이 SQLite 파일에 요구하는 것. **스키마 DDL은 쓰기 측이 확정**하되 아래를 만족해야 두 설계가 맞물린다. (단가표는 읽기 측 소유 - 계약에서 제외.)

**스냅샷 테이블 (`/api/oauth/usage`):**
1. **원본 JSON 전문 보존**(CLAUDE.md 3) - 텍스트 컬럼에 verbatim, 모르는 필드(`tangelo` 등) 포함.
2. **캡처 타임스탬프**(폴링 시각, UTC) - 단조 정렬·gap 검출용, 인덱스.
3. 조회용 파생: `five_hour_pct`, `weekly_all_pct`, `weekly_scoped_pct`, 각 `limits[]`의 `kind`/`is_active`/`scope.model.display_name`, 창별 `resets_at`. **파생 컬럼으로 제공하거나, 그게 없으면 원본 JSON이 안정적으로 존재해 읽기 측이 `json_extract`할 수 있음을 보장.** (읽기 측은 인덱스 위해 파생 컬럼 선호.)

**이벤트 테이블 (트랜스크립트, M5/M6):**
4. 이벤트당: **이벤트 타임스탬프(UTC)** - 시간창·단가 effective_from 조회용; `model` id; `input_tokens`, `output_tokens`, `cache_read_input_tokens`, `cache_creation.ephemeral_{5m,1h}_input_tokens`, `service_tier`.
5. **귀속 키:** **`cwd`(또는 그 안정 해시)** - 프로젝트별 집계에 필수; `gitBranch`(선택); `isSidechain`(서브에이전트 귀속); `sessionId`·`requestId`(dedup·멱등); **`source`/adapter id**(WSL/Windows - 완료기준4).
6. **중복 제거 책임 명시:** 완료기준4는 중복 `sessionId` 제거를 요구한다. **쓰기 측이 dedup(고유 인덱스)해 읽기 측이 신뢰하고 SUM만** 하도록 하거나, 그렇지 않으면 읽기 측이 dedup할 수 있는 키(예: `requestId` 또는 `sessionId`+메시지 식별자)를 보장. **전자 권장.**
7. **"billable 토큰" 정의 소재:** 완료기준3은 "billable 토큰 합"을 요구. 무엇이 billable인지(입력+출력? 캐시 포함? 가중치?)의 단일 정의가 필요. **원본 구성 토큰(위 4의 컬럼들)을 그대로 제공 → 읽기 측이 정의를 코드로 소유·테스트(오라클 대조).** 정의를 쓰기 측에 두지 말 것(테스트 독립성 위해).

**일반:**
8. **DB 파일이 읽기 전 존재해야 함**(검증: `readOnly` 열기는 없는 파일에 실패). 쓰기 측이 첫 구동 시 파일+스키마를 생성. 읽기 측은 "DB 아직 없음"을 정의된 로딩/빈 상태로 처리하되, **정규 경로**와 "읽기전용 열기가 파일을 생성하지 않음"을 안다.
9. **WAL은 쓰기 측이 설정**(001 실측 동시성). 읽기 측은 읽기전용으로 열며 checkpoint 안 함(수용).
10. **타임스탬프 규약 단일화** - 전부 UTC, ISO-8601 또는 epoch 중 하나로 일관.
11. **스키마 버전 노출** - `PRAGMA user_version` 또는 메타 테이블로. 읽기 측이 비호환 스키마를 감지.
12. **DDL ↔ `shared` 타입 동기** - 쓰기 측이 DDL을 바꾸면 `shared`의 행 타입도 갱신(이것이 계약 그 자체). 축 4의 `shared` 패키지가 이 동기의 단일 지점.

## 반증 조건 (이 설계가 틀릴 수 있는 경우)

- **cwd 귀속이 불안정:** 트랜스크립트가 `cwd`를 신뢰성 있게 담지 못하면(누락/회전) SQL project GROUP BY 설계가 깨짐 → 귀속 키 재설계(쓰기 측과 협의).
- **cross-package raw-.ts 스트립 실패:** `shared`를 raw `.ts`로 import할 때 Node 스트립이 안 먹으면 → `shared`만 `tsc` 빌드(폴백) 또는 단일 패키지로 축소.
- **`output: export`가 필요한 동적/인터랙션 뷰를 못 담음:** → 소형 서버 배포(축3 차선 a)로 후퇴, "0 실데이터" 보장 재점검.
- **연 20만 행 가정 붕괴:** 읽기전용 단일 파일이 느려지면 → 저장 재검토(쓰기 측 동반).
- **단가 개정이 잦고 복잡해짐(다수 tier/effective_from):** → 단가표를 TS 상수에서 커밋 JSON으로(축2 차선 c).
- **빌드타임 합성 sqlite 생성이 위험 판정:** → 데모 프로바이더가 JSON을 TS로 직접 집계(축5/3 대안).
